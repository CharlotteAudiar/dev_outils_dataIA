# Architecture

Décisions techniques (ADR) et schémas d'infrastructure du projet.

## Framework d'orchestration candidats

**Cadrage retenu** : le développement se concentre d'abord sur les cas d'usage 1 (connaissance des données Audiar / catalogue), 2 (extraction depuis une table en base), 3 (manipulation Excel assistée) et 5 (géomatique/QGIS) de l'analyse fonctionnelle. Le framework doit fonctionner en **mode projet** : accès à un dossier de travail complet, exécution autonome de tâches multi-étapes, appel d'outils MCP (QGIS, Postgres, Excel, filesystem) — sur le modèle de Claude Cowork/Claude Code, plutôt qu'une simple interface de chat avec appel d'outil ponctuel.

**Note de méthode** : l'évaluation ci-dessous ne s'appuie plus sur les outils mentionnés dans les fiches cas d'usage de l'analyse fonctionnelle (`draft/2026_IA&BD_Analyse fonctionnelle_V1.txt`) — ces mentions reflétaient les outils déjà testés empiriquement par les chargés d'études au moment des entretiens, pas une évaluation comparative. Le travail de recherche et de comparaison des frameworks commence ici ; chaque candidat est jugé sur ses caractéristiques propres (mode projet, MCP, contrôle des opérations, souveraineté, compétences requises, pérennité).

Recherche limitée aux frameworks **open source et gratuits**, relevée le 17/07/2026 (complétée le même jour avec deux candidats additionnels : VS Code + Cline, et Eigent).

| **Compétences nécessaires** | Installation/usage de base sans compétence dev (binaire Desktop ou script d'installation) ; personnalisation avancée (recipes) via YAML, pas de code obligatoire ; contribuer au cœur demande du Rust. | Déploiement simple (Docker ou app Desktop clé en main), accessible à un profil admin système basique ; configuration des agents/skills en no-code ; contribuer/étendre demande du JavaScript (Node.js). | Déploiement via Docker (une commande), accessible à un profil admin système basique ; le mode agent/MCP nécessite de configurer le proxy `mcpo` (un peu de ligne de commande) ; contribuer demande du Python et du Svelte. | Déploiement via Docker Compose avec plusieurs services à orchestrer (MongoDB, Redis, MeiliSearch, pgvector) — demande une compétence admin système/Docker plus confirmée que les autres candidats ; contribuer demande du JavaScript/TypeScript. | Installation simple (extension VS Code standard), aucune compétence dev requise pour l'usage de base ; mais l'usage quotidien se fait dans un éditeur de code (arborescence de fichiers, terminal visibles) — suppose un minimum de culture technique côté utilisateur ; contribuer à Cline demande du TypeScript. | Déploiement via app Desktop (installeur) ou Docker, accessible sans compétence dev pour l'usage de base ; configurer plus finement l'orchestration multi-agents (rôles, parallélisation) demande davantage de compétence côté administration ; contribuer demande du Python et du TypeScript. |
| **Facilité de prise en main** | **Moyenne à difficile** — usage par défaut en ligne de commande (CLI), peu naturel pour un non-développeur ; le Desktop app simplifie l'accès mais reste un outil "assistant technique" sans onboarding guidé ; convient surtout aux profils avancés. | **Facile** — interface de chat familière type ChatGPT, onboarding guidé (création de workspace, dépôt de documents) ; le mode `@agent` demande un peu d'appropriation mais reste accessible aux profils intermédiaires. | **Facile** pour le chat/RAG courant (interface proche de ChatGPT, prise en main immédiate) ; la configuration MCP/agentique reste du ressort de l'administrateur, transparente une fois en place pour l'utilisateur final. | **Facile** pour l'usage courant (autre clone ChatGPT, prise en main immédiate) ; l'utilisateur final choisit un agent préconfiguré dans une liste, sans avoir à gérer les outils/MCP lui-même. | **Difficile pour un profil généraliste** — nécessite de se familiariser avec un éditeur de code (arborescence, terminal, panneau de diff) avant même d'utiliser Cline ; plus naturel pour les profils avancés déjà à l'aise avec un IDE. | **Moyenne** — interface Desktop pensée pour l'utilisateur final (à la Cowork), mais le concept de "workforce" multi-agents (assigner des rôles, superviser plusieurs agents en parallèle) est nouveau et demande un temps d'appropriation supérieur à un simple chat ; documentation/communauté encore limitée vu la jeunesse du projet. |
| **Code** | Rust (cœur) + TypeScript (interface Desktop Electron) | JavaScript/Node.js (serveur Express) + React (frontend Vite) | Python/FastAPI (backend) + Svelte (frontend) | JavaScript/TypeScript (Node/Express + React) + MongoDB | TypeScript (extension + interface webview React) | Python (backend, s'appuie sur CAMEL-AI) + TypeScript (frontend) |
| **Chances de rester gratuit/open source** | **Élevé** — gouvernance transférée à une fondation neutre (Linux Foundation, déc. 2025), structurellement moins encline à fermer la licence qu'un éditeur unique ; code Apache 2.0 déjà publié, de toute façon irrévocable. | **Moyen** — MIT, mais startup à capital-risque (YC) sous pression de monétisation ; le code déjà publié reste MIT, mais risque que les futures fonctionnalités avancées basculent vers une offre cloud/entreprise payante plutôt que d'enrichir le dépôt open source. | **Faible-moyen** — précédent concret défavorable : passage en 2025 d'une licence permissive à la "Open WebUI License" (non-OSI), au moment même où une offre Enterprise apparaissait ; tendance déjà amorcée vers plus de restrictions. | **Élevé sur la licence** (MIT, pas de capital-risque poussant à la fermeture) mais incertitude sur la continuité de la maintenance (mainteneur unique) — distinguer "reste gratuit/ouvert" (probable) de "reste activement maintenu" (moins certain). | **Moyen** — VS Code (Microsoft) reste gratuit sans ambiguïté depuis plus de 10 ans, signal de continuité fort ; Cline (32M$ levés) a la même dynamique qu'AnythingLLM : code Apache 2.0 déjà publié irrévocable, mais risque que les futures fonctionnalités soient réservées à une offre entreprise déjà évoquée. | **Faible-moyen** — startup jeune (~8 personnes) et financée par capital-risque ; les fonctions "entreprise" (SSO, RBAC, audit) sont présentées comme incluses aujourd'hui, mais à surveiller si elles basculent vers une édition payante à mesure que l'éditeur cherche à générer du revenu ; recul insuffisant (projet pré-v1.0) pour juger de la trajectoire. |
| **Chances qu'un fork soit créé et maintenu open source (si la licence/le prix change)** | **Élevé** — Apache 2.0 permissif, mais surtout risque déclencheur déjà faible (gouvernance en fondation) ; si le cas se présentait, l'écosystème Linux Foundation a un précédent direct et réussi : Valkey, le fork communautaire de Redis créé après le changement de licence de Redis Inc. en 2024, aujourd'hui bien maintenu sous cette même fondation. Bémol : bassin de contributeurs Rust plus restreint que JS/Python, ce qui pourrait ralentir un fork si besoin. | **Moyen** — MIT et large communauté (54k ★) rendent un fork techniquement facile, mais aucune crise de licence ne s'est encore produite pour tester cette hypothèse en pratique ; à surveiller si Mintplex Labs venait à restreindre l'accès aux futures fonctionnalités. | **Cas déjà documenté** — le changement de licence d'avril 2025 a réellement déclenché des discussions actives de fork (repartant de la dernière version BSD), la communauté citant elle-même les précédents réussis de Valkey (fork de Redis) et OpenSearch (fork d'Elasticsearch par AWS) ; cela dit, à ce jour aucun fork dominant et pérenne d'Open WebUI ne s'est encore imposé — des discussions, pas encore un projet abouti. | **Moyen-élevé** — MIT, projet important (33k ★) ; le risque principal ici est l'abandon plutôt qu'une fermeture de licence (pas de capital-risque poussant à restreindre), et les forks de projets abandonnés sont un schéma courant et généralement viable pour un dépôt de cette taille. | **Élevé — le précédent le plus solide du tableau.** VS Code dispose déjà d'un fork mature et activement maintenu depuis des années : VSCodium, qui retire la télémétrie et les éléments propriétaires de Microsoft et s'appuie sur le registre d'extensions ouvert Open VSX (Eclipse Foundation) — la preuve concrète que ce schéma fonctionne dans la durée. Côté Cline, l'écosystème a montré une forte propension à forker (Roo Code, puis Kilo Code fusionnant Cline + Roo Code), même si la pérennité de ces forks reste inégale (Roo Code a été archivé en mai 2026). | **Faible-moyen** — Apache 2.0 permissif en théorie, mais communauté encore jeune et restreinte (14k ★, projet pré-v1.0) ; un fork demanderait une masse critique de contributeurs motivés qui n'existe pas encore clairement à ce stade. |
| **Avantages** | Le plus proche du fonctionnement Cowork ; ~30k ★, 350+ contributeurs, gouvernance Linux Foundation (pérennité) ; compatible avec tout LLM (dont modèles locaux via Ollama) — bon point pour la souveraineté ; workflows partageables en YAML ("recipes"). | Application Desktop clé en main (installation simple, moteur d'inférence embarqué) ; ~54k ★, un des projets self-hosted les plus populaires ; RAG intégré nativement ; workspaces isolés par projet limitent les fuites de contexte. | Excellent pour RAG/catalogue documentaire ; très large communauté, déploiement self-hosted simple (Docker) — bon point souveraineté. | ~33k ★, réellement multi-utilisateurs avec auth (adapté à un déploiement agence) ; licence MIT sans ambiguïté ; bonne intégration MCP documentée. | Mode "tout approuver" par défaut — la mise en œuvre la plus littérale de l'exigence de contrôle pas à pas ; ~58k ★, 4M+ développeurs ; compatible avec quasiment tous les fournisseurs de modèles dont le local (Ollama) — bon point souveraineté ; structure bien financée donc développement soutenu. | Positionnement le plus proche de Cowork parmi tous les candidats (revendiqué comme tel par l'éditeur) ; ~14k ★, 1,7k forks ; modèle-agnostique (Claude, GPT, Gemini, LLM local) — bon point souveraineté ; exécution locale, fonctions entreprise (SSO, RBAC, journaux d'audit) incluses. |
| **Inconvénients** | Pensé à l'origine pour un usage développeur (édition de code, shell) ; l'UI Desktop est plus simple que la CLI mais reste moins guidée qu'un outil grand public ; contrôle pas à pas des opérations moins mis en avant que chez d'autres candidats. | Le mode agent (accès fichiers, MCP) est une brique parmi d'autres de l'outil, moins mature/central que chez Goose ; permissions dossier globales (lecture+écriture sur tout le dossier autorisé), granularité plus grossière. | Licence gratuite mais plus "open source" au sens strict (OSI) depuis 2025 — point de vigilance si l'agence a une politique stricte sur les licences ; le mode agentique "mode projet" nécessite une couche supplémentaire (mcpo + configuration). | Conçu comme clone de ChatGPT (conversation) plutôt que comme agent de type "projet" ; moins orienté exécution autonome multi-étapes que Goose ; gouvernance mono-mainteneur, moins résiliente qu'une structure d'entreprise ou de fondation. | Interface intrinsèquement liée à un éditeur de code, peu adaptée aux profils généralistes ; pas une application "grand public" autonome, dépend de VS Code comme socle. | Projet encore jeune (version 0.0.90 au 15/04/2026, pas encore de v1.0), petite équipe (~8 personnes) — maturité et stabilité à vérifier en conditions réelles ; architecture multi-agents plus complexe à appréhender/configurer qu'un simple agent séquentiel. |

**Écarté du comparatif** : OpenHands (trop spécialisé développement logiciel/cloud, peu pertinent pour des chargés d'études non-développeurs) ; Roo Code, un fork de Cline pourtant bien noté, **archivé en mai 2026** (abandon de la maintenance) ; Kilo Code (fusion Cline + Roo Code), écarté à ce stade car produit récent porté par une startup financée, sans le recul de maturité des autres candidats — à surveiller.

**Piste à date** : deux candidats se distinguent nettement sur le critère "mode projet façon Cowork" — **Eigent**, qui revendique explicitement ce positionnement et est model-agnostique, mais dont la jeunesse du projet (version 0.0.x) appelle à la prudence ; et **VS Code + Cline**, déjà cité dans l'analyse fonctionnelle pour 3 des 4 cas d'usage retenus, avec le mécanisme de contrôle pas à pas le plus abouti, au prix d'une interface plus technique (éditeur de code). Goose reste une option solide et pérenne (gouvernance Linux Foundation) mais moins guidée pour des profils non-développeurs. Ces pistes sont à tester sur les cas d'usage 1, 2, 3 et 5 avant arbitrage.
